Vấn đề 1 — Giá stale (đã thấy với Bà Nà)
Giá thay đổi nhưng DB không biết. Mindtrip hiển thị 700k, thực tế 1M. Xảy ra với mọi nơi có vé có thu phí.
Vấn đề 2 — Place đóng cửa vĩnh viễn
Nhà hàng đóng, khách sạn chuyển chủ, địa điểm ngừng hoạt động. Nếu DB không biết → LLM recommend chỗ không tồn tại.
Vấn đề 3 — Missing / sai fields
lat/lng = null → geo clustering sai
hours = null → không validate opening hours
duration = 0 → slot scheduling sai
tags = [] → preference matching không hoạt động
Vấn đề 4 — Sai destination assignment
Hội An seeded với tourist_dest_id = "da-nang". Mỹ Sơn seeded với province = "Quảng Nam" thay vì tourist destination. Đây là lỗi thủ công khi seed.
Vấn đề 5 — Duplicate entries
Cùng 1 địa điểm được import từ nhiều nguồn → xuất hiện 2-3 lần trong kết quả. Ví dụ: "Hang Sửng Sốt", "Sung Sot Cave", "Động Sửng Sốt" → 3 records khác nhau.
Mỗi field có 3 thuộc tính: value + confidence + verified_at
Không lưu "giá 1 triệu" mà lưu "giá 1 triệu, scraped, hôm qua". Khi serve data, hệ thống biết field nào đáng tin, field nào cần cảnh báo.
-- Thay vì lưu flat: places { price_vnd INT, lat FLOAT, hours TEXT } -- Lưu với provenance: place_fields { place_id UUID, field_name TEXT, -- 'price_vnd', 'lat', 'hours', 'duration_min' field_value TEXT, -- serialized value confidence FLOAT, -- 0.0 → 1.0 source TEXT, -- 'manual_admin', 'scraped_official', -- 'scraped_tripadvisor', 'llm_inferred', 'user_report' verified_at TIMESTAMPTZ, verified_by TEXT -- admin username hoặc 'crawler-v2' } -- Composite quality score quality_score = AVG(confidence) OVER required_fields -- Required fields: name, tourist_dest_id, lat, lng, place_type -- place không được serve nếu quality_score < 0.5

Anomaly detection — quan trọng nhất
def sync_price(place, new_price): last = get_last_price(place.id) # Sanity check: thay đổi > 50% → không auto-accept if last and abs(new_price - last) / last > 0.5: create_review_task( place_id = place.id, field = 'price_vnd', old_val = last, new_val = new_price, reason = f"Price changed {((new_price-last)/last*100):.0f}%" ) return # Giữ giá cũ cho đến khi admin confirm # Bình thường → update và lưu snapshot upsert_price_snapshot( place_id = place.id, price_vnd = new_price, confidence = 0.8, # scraped, không phải manual source = 'official_site', verified_at = now() )
Stale warning trong response
Khi serve place có price verified_at > 30 ngày → thêm flag STALE_PRICE vào response. LLM enrichment node sẽ viết: "Giá vé có thể đã thay đổi, nên kiểm tra lại trước khi đặt."
3 nguồn feedback từ user
Flag wrong User báo sai giá, địa điểm đóng cửa, thông tin không đúng
Trip completed Sau chuyến đi, user confirm "đã đến, giá đúng" → tăng confidence
Click-through rate Place được recommend nhưng không click → signal quality thấp
Feedback xử lý như thế nào
-- Khi user flag "địa điểm này đóng cửa rồi" UPDATE places SET status = 'flagged', review_required = TRUE WHERE id = $place_id; -- Tự động: nếu >= 3 users flag cùng 1 vấn đề -- → SUSPEND place (ẩn khỏi recommendations) -- → tạo urgent review task cho admin SELECT place_id, COUNT(*) as flag_count FROM user_flags WHERE created_at > NOW() - INTERVAL '7 days' GROUP BY place_id HAVING COUNT(*) >= 3; -- → suspend + notify admin -- Khi user confirm "đã đến, thông tin đúng" UPDATE place_fields SET confidence = LEAST(confidence + 0.05, 1.0), verified_at = NOW(), verified_by = 'user_confirmed' WHERE place_id = $place_id AND field_name = 'is_open';
Confidence decay — quan trọng nhất
Confidence KHÔNG tự nhiên giữ nguyên. Nó decay theo thời gian nếu không có verification mới.
-- Chạy daily job UPDATE place_fields SET confidence = confidence * 0.98 -- decay 2%/ngày WHERE verified_at < NOW() - INTERVAL '30 days' AND source !='manual_admin' ; -- Manual admin data decay chậm hơn (0.5%/ngày) -- Kết quả: sau 6 tháng không verify, confidence tự xuống ~30% -- → tự động vào review queue khi < 0.4
Không có hệ thống nào giữ data sạch mãi mãi mà không cần người. Admin review queue là bắt buộc. Mục tiêu là tối thiểu hóa manual work, không phải loại bỏ hoàn toàn.
